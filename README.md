## Overview

This project detects water pipe leaks in real time by combining IoT sensor hardware (ESP32 + flow/pressure sensors) with machine learning models trained on location-aware GIS data. Sensor readings are published over MQTT, processed by a Node.js backend, and classified by one of three ML models. Alerts are broadcast instantly to a live web dashboard.

The system was designed in the context of Rwandan water utilities, where Non-Revenue Water (NRW) losses are a significant operational challenge.

## Machine Learning Notebook

### What it does

The notebook (`leakdetection.ipynb`) runs a full supervised and unsupervised leak-detection pipeline:

| Step | Description |
|---|---|
| Data loading | Reads the 5,000-row GIS-annotated dataset |
| EDA | Class distribution charts, Pressure vs Flow scatter, correlation heatmap |
| Feature engineering | `Pressure_Flow_Ratio`, `Pressure_x_Vib`, `Flow_Temp_Ratio` |
| Preprocessing | Label encoding, MinMaxScaler, stratified train/test split |
| SMOTE | Balances the 6.5% minority leak class before training |
| Model 1 — Random Forest | 300 trees, balanced class weight, feature importance chart |
| Model 2 — Neural Network | 4-layer MLP, BatchNorm, Dropout, EarlyStopping on val_recall |
| Model 3 — Isolation Forest | Unsupervised anomaly detection, no labels required |
| Evaluation | ROC curves, Precision-Recall curves, confusion matrices, summary table |
| Export | Saves `.pkl`, `.keras`, and `.json` files used by the backend |

## Backend Application

### Architecture

```
ESP32 / simulator.js
    │  JSON payload every 5 s
    │  topic: iot/sensor/water
    ▼
MQTT Broker (HiveMQ public / local Mosquitto)
    ▼
index.js  (Node.js · Express · Socket.io)
    │  spawns Python child process per reading
    ▼
inference.py
    │  loads rf_leak_detector.pkl + feature_scaler.pkl
    │  engineers derived features
    │  RandomForest.predict_proba() → JSON result
    ▼
index.js
    ├── Publishes result  → iot/results
    ├── If LEAK: publishes alert → iot/alerts/leak
    ├── Appends entry to alerts.log
    └── Broadcasts to dashboard via Socket.io
```

### Quick Start

**1. Train the models (if not already done)**

Run the notebook end-to-end. This produces `rf_leak_detector.pkl`, `feature_scaler.pkl`, and `feature_columns.json` in `D:\Leak\`.

**2. Install Node.js dependencies**

```powershell
cd D:\Leak\backend
npm install
```

**3. Configure the broker** (optional)

Edit `.env` — the default uses the free public HiveMQ broker, which is fine for development:

```env
MQTT_BROKER=mqtt://broker.hivemq.com
MQTT_TOPIC_SENSOR=iot/sensor/water
MQTT_TOPIC_RESULT=iot/results
MQTT_TOPIC_ALERT=iot/alerts/leak
LEAK_THRESHOLD=0.5
PORT=3000
```

For local testing with Mosquitto:
```env
MQTT_BROKER=mqtt://localhost:1883
```

**4. Start the backend**

```powershell
node index.js
```

**5. Run the simulator** (no ESP32 needed)

Open a second terminal:

```powershell
node simulator.js
```

Sends a live reading every 5 seconds. Every 10th reading simulates a leak.

**6. Open the dashboard**

```
http://localhost:3000
```

## ESP32 Hardware

### Sensor Payload

The ESP32 firmware should publish this JSON to `iot/sensor/water`:

```json
{
  "Pressure": 62.3,
  "Flow_Rate": 78.1,
  "Temperature": 97.2,
  "Vibration": 2.8,
  "RPM": 2050,
  "Operational_Hours": 5000,
  "Latitude": -1.9441,
  "Longitude": 30.0619,
  "Zone_enc": 2,
  "Block_enc": 1,
  "Pipe_enc": 3
}
```

`Zone_enc`, `Block_enc`, `Pipe_enc` are integer-encoded zone IDs (0-indexed, matching the label encoding used during training).

### Wiring

```
YF-B5 Flow Sensor
  VCC    → ESP32 5V (VIN)
  GND    → ESP32 GND
  Signal → ESP32 GPIO 34

Pressure Transducer (0–1.2 MPa, 0.5–4.5 V output)
  VCC    → 5V
  GND    → GND
  Signal → Voltage divider → ESP32 GPIO 35

  Voltage divider (5V → 3.3V):
    Signal ──┬── 10 kΩ ── GPIO 35
             └── 20 kΩ ── GND

Solenoid Valve (12V, normally closed)
  Relay IN  → ESP32 GPIO 26
  Relay VCC → 5V
  Relay COM → 12V+
  Relay NC  → Valve+
  Valve−    → 12V GND
```

## Leak Threshold Tuning

Set `LEAK_THRESHOLD` in `.env` to control sensitivity:

| Value | Behaviour |
|---|---|
| `0.3` | High sensitivity — fewer missed leaks, more false alarms |
| `0.5` | Balanced (default) |
| `0.7` | Conservative — fewer false alarms, more missed leaks |

For a water utility, **0.3–0.4** is recommended — a missed leak costs more in NRW revenue loss than a false alarm costs in inspection time.

## Key Findings

- `Pressure` and `Flow_Rate` are the two strongest individual leak predictors, validating the YF-B5 + pressure transducer hardware choice.
- `Pressure_Flow_Ratio` (engineered feature) captures the hydraulic relationship described by Oren & Stroh (2013) and consistently ranks in the top 3 feature importances.
- SMOTE is essential at 6.5% leak prevalence — without it, models default to predicting "Normal" for nearly all samples.
- **Random Forest** is recommended for production deployment in the Node.js backend due to its low inference latency and interpretability.
- **Neural Network** achieves the highest ROC-AUC but requires a Python runtime and longer inference time.
- **Isolation Forest** requires no labels, making it ideal for new pipe installations with no historical leak data.
